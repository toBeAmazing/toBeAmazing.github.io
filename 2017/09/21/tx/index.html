学习笔记————事务隔离级别

发表于 2017-09-21 | 分类于 学习笔记 | | 本文总阅读量 次

4种隔离级别

标准SQL规范中的定义了以下4个事务隔离级别：

  1. Read uncommitted
    一个事务可以读取到另一个未提交事务的数据
  2. Read committed
    一个事务要等另一个事务提交后才能读取数据
  3. Repeatable read
    一个事务进行读操作时，再事务提交前，其他事务不能修改第一个事务读取的内容
  4. Serializable
    要求事务序列化执行，不能并发执行

实践

read uncommitted

  1. 修改事务隔离级别为read uncommitted

  2. 新建一个连接，使用默认隔离级别

事务一(READ-UNCOMMITTED) 事务二(REPEATABLE-READ)
START TRANSACTION; START TRANSACTION;
SELECT data FROM tx WHERE id = 1;
结果为1
UPDATE tx SET data = ‘111’ WHERE id = 1;
SELECT data FROM tx WHERE id = 1;
再次查询结果为111
commit; rollback;

由于事务1的隔离级别为读未提交，再以上过程中，第二个SELECT语句读取到了事务二未提交的值，这种现象为脏读。

Read committed

同样修改事务一隔离级别为read committed 事务2使用默认隔离级别

事务一(READ-COMMITTED) 事务二(REPEATABLE-READ)
START TRANSACTION; START TRANSACTION;
SELECT data FROM tx WHERE id = 1;
UPDATE tx SET data = ‘111’ WHERE id = 1;
commit;
SELECT data FROM tx WHERE id = 1;
commit;

在以上过程中，事务1 两次读取的同一行数据产生了不一致的情况，也就是不可重复度。在这种隔离级别下，可以避免脏读的发生，但不能解决不可重复读。

Repeatable reads

事务一(REPEATABLE-READ) 事务二(REPEATABLE-READ)
START TRANSACTION; START TRANSACTION;
SELECT data FROM tx WHERE id = 1;
commit;
其他事务无法在此事务提交前修改此事务读取的数据，解决了不可重复读的问题
UPDATE tx SET data = ‘111’ WHERE id = 1;
commit;

只有事务一提交后，事务二才能执行更改该行数据的操作。因此，在事务一开始到结束期间，读取多少次该行数据，结果都一样。
但是这种隔离级别无法解决幻读的情况。

幻读

事务一(REPEATABLE-READ) 事务二(REPEATABLE-READ)
START TRANSACTION; START TRANSACTION;
SELECT * FROM tx WHERE data in (‘1’,’2’,’3’,’4’);
INSERT INTO tx VALUES (11,’1’);
commit;
SELECT * FROM tx WHERE data in (‘1’,’2’,’3’,’4’);
commit;

事务一第一次查询，会返回4条记录，其他事务无法修改这四条记录。但是事务二可以执行插入语句，因此当事务一再次查询时，会得到5条记录。事务一的两次查询结果不一致，这就是幻读。

Serializable

最高隔离级别。事务一在读取表中记录时，事务二不能对A表做增、删、改的操作。事务一在更新表中记录时，事务二不能对该表中的所有记录做任何操作。
